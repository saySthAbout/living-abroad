## 1. 개요 (Overview)

사용자의 나이, 학력, 경력, 직업, 언어 능력, 자금, 선호 지역 등 개인 프로필을 입력받아, 이민·취업·유학 가능성이 높은 국가와 비자 유형을 추천하고 추천 근거·준비 절차·공식 정책 기반 상담을 제공하는 AI 플랫폼이다.

핵심 원칙은 **"개인의 비자 승인 여부를 예측하지 않는다"**는 것이다. 공개 통계만으로는 실제 비자 심사 결과(개인별 비공개 사유, 정책적 판단 포함)를 정확히 예측할 수 없으므로, 본 서비스는 승인 확률이 아닌 **적합도(fit score)** 를 제공하는 참고용 정보 서비스로 포지셔닝한다.

## 2. 문제 정의 및 배경

- 해외 이주를 고려하는 사람들은 국가별 비자 제도, 자격 요건, 노동시장 상황이 제각각이라 정보를 종합적으로 비교하기 어렵다.
- 공식 이민청 자료는 국가별로 형식이 다르고(PDF, 다국어, 비정형 표), 개인이 자신의 조건에 맞는 경로를 파악하기 어렵다.
- 반면 개인별 비자 승인 여부를 예측하는 서비스는 데이터 부재와 오남용 리스크로 인해 신뢰할 수 없고 위험하다.
- 따라서 "승인 확률 예측"이 아닌 "조건 기반 적합도 추천 + 공식 정책 상담"이라는 좁고 신뢰 가능한 문제로 범위를 좁힌다.

## 3. 목표 (Goals)

### 프로젝트 목표
- ML(적합도 랭킹) · DL(경력-직업 임베딩) · LLM(RAG 상담)을 모두 자연스럽게 포함하는 포트폴리오급 결과물 제작
- Vue 3 – Spring Boot – FastAPI로 이어지는 실무형 마이크로서비스 구조 구현
- "정답 라벨이 없는 추천 문제"를 규칙 기반 + ML 보정이라는 현실적 방식으로 해결한 사례 제시

### 비목표 (Non-goals)
- 개인의 실제 비자 승인/거절 여부 예측
- 전 세계 모든 국가 지원 (MVP는 3개국)
- 법률 자문 제공

## 4. 타겟 사용자

| 페르소나 | 설명 |
|---|---|
| 해외 취업 준비자 | 개발자 등 전문직으로 캐나다/호주/영국 취업 이민을 고려 |
| 유학 후 이민 희망자 | 유학을 거쳐 영주권까지 고려하는 사용자 |
| 이민 초기 탐색자 | 아직 국가를 정하지 못하고 여러 나라를 비교하는 사용자 |

## 5. 범위 (Scope)

### 5.1 지원 국가 (MVP)
캐나다, 호주, 영국 — 데이터 확보 용이성과 팀 구현 가능성을 고려해 3개국으로 제한한다.

### 5.2 In Scope
- 회원가입/로그인, 이민 프로필 입력
- 3개국 적합도 추천 및 비자 유형 추천
- 요구조건 대비 사용자 조건 비교, 부족 조건 분석
- 국가별 생활비·고용·이민 통계 비교
- RAG 기반 공식 정책 AI 상담
- 준비 체크리스트 및 진행률 대시보드
- 관리자: 국가/비자/정책 문서 관리, 데이터셋·모델 버전 관리, 오답 신고 처리

### 5.3 Out of Scope (확장 기능으로 후순위)
- 이민 서류 OCR
- 다국어 번역
- 부족 직업군 자동 매칭 고도화
- 정책 변경 알림, 상담 기록 요약
- 가족 구성원 포함 추천

## 6. 사용자 시나리오 (User Flow)

1. 사용자가 회원가입 후 이민 프로필(나이, 국적, 학력, 전공, 경력, 언어점수, 자금, 가족 동반 여부, 선호 국가/기후/생활비, 목적)을 입력한다.
2. 시스템이 규칙 기반 비자 자격 필터를 적용해 명백히 부적합한 경로를 제외한다.
3. ML 모델이 국가별 적합도 점수를 계산하고 순위를 매긴다.
4. DL 임베딩 모델이 사용자 경력과 국가별 부족 직업군의 의미적 유사도를 계산해 보정한다.
5. RAG가 관련 공식 정책 문서를 검색하고, LLM이 사람이 이해하기 쉬운 추천 사유·준비 절차 설명을 생성한다.
6. 사용자는 결과 화면에서 국가 순위, 강점/보완점, 예상 준비 기간·비용, 공식 출처, 기준일을 확인한다.
7. 필요 시 AI 챗봇에게 후속 질문을 하면, 챗봇은 RAG로 검색된 공식 문서 범위 내에서만 답변하고, 근거가 없으면 답변을 보류하고 공식 페이지 확인을 안내한다.
8. 사용자는 준비 체크리스트를 저장하고 진행률을 관리한다.

## 8. AI/ML/DL/LLM 요구사항

### 8.1 규칙 기반 1차 필터 + 점수화
연령/학력/경력/직업수요/언어능력/예산 6개 항목에 가중치를 부여한 100점 만점 규칙 기반 점수를 1단계로 산출한다 (예: 연령 20, 학력 15, 경력 20, 직업수요 20, 언어 15, 예산 10).

### 8.2 ML — 국가 적합도 순위 보정
- 정답 라벨(개인별 최적 국가)이 존재하지 않으므로, 국가별 취업률·임금·정착률·유입 추세 등 객관적 통계로 "국가 성공 가능성/정착 환경 점수"를 학습해 규칙 기반 점수를 보정한다.
- 후보 모델: Logistic Regression, Random Forest, XGBoost, LightGBM, Learning to Rank, K-Means(사용자 유형 군집화)
- 장기적으로 추천 만족도, 실제 선택 국가/비자, 준비 시작 여부 등 사용자 피드백을 학습에 반영한다.

### 8.3 DL — 경력·직업 의미 유사도
- Sentence Transformer 계열 임베딩으로 사용자의 실제 경력 서술과 국가별 부족 직업군 분류를 의미적으로 매칭한다.
- 정형 데이터만으로 충분한 다른 기능에는 DL을 억지로 적용하지 않는다 (표 형태 데이터는 XGBoost/LightGBM이 더 적합할 수 있음을 인지).

### 8.4 LLM — RAG 기반 정책 상담
- 파이프라인: 사용자 질문 → 질문 임베딩 → Vector DB 검색 → 관련 공식 문서 추출 → LLM 답변 생성 → 출처·기준일 표시
- 답변 구조: 가능한 경로 → 현재 조건의 강점 → 부족한 요건 → 추가 확인 사항 → 공식 출처 → 정보 기준일 → 법률 자문이 아니라는 안내
- **LLM은 추천 순위를 직접 산출하지 않는다.** 순위는 규칙+ML+DL 결과이며, LLM은 그 결과를 설명하는 역할만 담당한다.

## 9. 데이터 요구사항

### 9.1 확보 가능한 데이터
UN International Migrant Stock, OECD International Migration Database/DIOC, Eurostat Migration and Asylum Data, 캐나다 IRCC 공개 데이터, 영국 Immigration System Statistics, 호주 Department of Home Affairs Visa Statistics, 미국 국무부 Visa Statistics 등 국가별 이민 통계, 노동시장 결과, 비자 신청/발급/거절 건수, 임금·실업률·물가·주거비, 정착 지표.

### 9.2 부족한 데이터 (사용하지 않음)
개인 신청자 상세 프로필, 제출 서류 내용, 개인별 승인/거절 결과, 심사관 판단 근거, 개인별 거절 사유. → 이 데이터가 없다는 전제 하에 설계하며, 개인별 승인 예측 기능은 만들지 않는다.

### 9.3 정책 데이터 관리 원칙
정책(점수 기준, 언어점수 기준, 최소 급여, 신청 비용, 부족 직업군, 비자 명칭 등)은 코드에 하드코딩하지 않고 DB에서 버전 관리한다.

필수 컬럼: `visa_program`, `visa_requirement`, `policy_document`, `policy_version`, `effective_from`, `effective_to`, `source_url`, `last_verified_at`

## 10. 시스템 아키텍처 / 기술 스택

```
Vue 3 (사용자·관리자 화면)
  ↓
Spring Boot (메인 백엔드: 인증, 프로필, 추천 이력, 체크리스트, 관리자 API)
  ├─ PostgreSQL (사용자·국가·비자·정책 데이터)
  └─ FastAPI (AI 서버: ML/DL/LLM 추론)
       ├─ ML 모델 (Scikit-learn, LightGBM/XGBoost)
       ├─ 임베딩 모델 (Sentence Transformers, PyTorch)
       ├─ Vector DB (pgvector 또는 Qdrant)
       └─ LLM API + LangChain/LlamaIndex(선택)
```

| 레이어 | 스택 |
|---|---|
| Frontend | Vue 3, TypeScript, Vite, Pinia, Vue Router, Axios, Chart.js/ECharts |
| Backend (메인) | Spring Boot, Spring Security/JWT, PostgreSQL |
| AI 서버 | FastAPI, Scikit-learn, LightGBM/XGBoost, Sentence Transformers, PyTorch |
| 인프라 | Redis(캐시/비동기 작업), Docker, GitHub Actions |

원칙: Vue는 Spring Boot만 호출하고, Spring Boot가 필요 시 FastAPI를 호출하는 단방향 구조를 유지해 관리 복잡도를 낮춘다.

## 11. 비기능 요구사항 (Non-Functional Requirements)

- **신뢰성/정확성**: RAG 검색 결과가 없으면 답변을 중단하고 공식 페이지 확인을 유도한다. 공식 정부 문서를 우선 사용하며, 숫자·자격 조건은 원문과 대조 후 게시한다.
- **투명성**: 모든 추천 결과에 근거, 공식 출처, 정보 기준일을 표시한다. 적합도 점수(예: 82점)는 승인 확률이 아님을 화면에 명확히 고지한다.
- **보안**: 회원 인증/인가는 Spring Security + JWT로 처리하며, 개인정보(자금, 경력 등 민감정보 포함)는 암호화 저장을 원칙으로 한다.
- **성능**: 추천 결과는 규칙 기반 → ML → DL → RAG/LLM 순으로 처리되며, 사용자 체감 응답 시간을 고려해 비동기 처리(Redis)를 활용한다.
- **유지보수성**: 정책 데이터는 하드코딩 금지, DB 버전 관리로 정책 변경에 대응한다.
- **법적/윤리적 고지**: 서비스 내 "이민 합격 보장", "영주권 승인 확률", "이 조건이면 무조건 승인", "AI 법률 상담", "공식 이민 심사 결과" 등의 표현을 금지한다. 하단에 다음 안내문을 상시 노출한다.

  > 본 서비스는 공개 데이터 기반의 참고용 정보와 적합도 분석을 제공합니다. 비자 승인 여부를 보장하지 않으며, 최종 신청 전 해당 국가의 공식 기관 또는 자격을 갖춘 전문가에게 확인해야 합니다.

## 13. 리스크 및 완화 방안

| 리스크 | 완화 방안 |
|---|---|
| 정답 라벨 부재로 ML 성능 검증이 어려움 | 규칙 기반 점수를 1차 baseline으로 사용, ML은 "최종 판단"이 아닌 "순위 보정"으로 역할 한정 |
| LLM 환각으로 잘못된 정책 정보 제공 | RAG 근거 없으면 답변 중단, 공식 문서 우선, 확정적 승인 표현 금지, 관리자 검수 기능 |
| 정책이 자주 바뀌어 정보가 outdated 됨 | 정책 DB 버전 관리(`effective_from/to`, `last_verified_at`), 관리자가 주기적으로 갱신 |
| 국가마다 데이터 구조가 상이함 | 원본 테이블 + 국가 공통 표준 테이블 + ETL 변환 계층 분리 |
| 크롤링 실패(HTML 변경, PDF, JS 렌더링 등) | 실시간 전체 크롤링 대신 공식 CSV/Excel/API와 관리자 수동 등록 문서를 우선 사용 |
| 법적 리스크(무자격 법률 자문으로 오인) | 매 답변에 "법률 자문 아님" 고지, 금지 표현 리스트 운영, 서비스 하단 면책 문구 상시 노출 |

## 14. 마일스톤 (제안)

| 단계 | 내용 |
|---|---|
| 1단계 | 데이터 수집·정제(3개국), DB 스키마 설계(정책 버전 관리 포함) |
| 2단계 | 규칙 기반 점수 엔진 + Spring Boot/FastAPI 기본 연동 |
| 3단계 | ML 적합도 모델 학습 및 순위 보정 적용 |
| 4단계 | DL 임베딩(경력-직업 유사도) 적용 |
| 5단계 | RAG 파이프라인 구축(벡터 DB, 정책 문서 색인, LLM 응답 생성) |
| 6단계 | Vue 프론트엔드(사용자/관리자 화면), 통계 시각화 |
| 7단계 | 통합 테스트, 환각 방지 안전장치 점검, 발표 준비 |
